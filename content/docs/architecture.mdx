Lupin is one ROS 2 Humble workspace split into **nine packages**. Each owns a single
concern, and they connect through a small set of typed topics and services rather than
shared state. The *same logic runs in Gazebo and on the real MIRTE Master V2*, only the
bring-up files and a single drive topic differ. This page is the map: who owns what, how
data flows from an empty map to a live world model, and where the sim/hardware line is
drawn.

## The nine packages

| Package | Responsibility |
| --- | --- |
| `lupin_msgs` | The interface contract: 10 messages + 6 services (`ament_cmake` / `rosidl`). No nodes. Everything else links against it. |
| `lupin_bringup` | Top-level launch per scenario (`sim_full.launch.py`, `hardware.launch.py`, `onboard.launch.py`, `cameras.launch.py`), the greenhouse-world generator, systemd units, and DDS-over-WiFi glue. |
| `lupin_navigation` | Nav2 + `slam_toolbox` tuned for the mecanum base: MPPI controller, `motion_model: Omni`, AMCL-vs-SLAM modes, and the `/lupin/nav/clear_map` reset service. |
| `lupin_mission` | The mission orchestrator: a hierarchical state machine driving `InspectionMission` and `ExplorationMission` over Nav2 + the greenhouse bridge. Publishes every tag scan as an `Observation`. |
| `lupin_perception` | Vision: `tag_annotator` (OpenCV ArUco AprilTags), `yolo_detector` (Ultralytics tulip/bug classifier), and `perception_aggregator` fusing them into discovered tags + flower observations. |
| `lupin_greenhouse_bridge` | A ROS wrapper around the course `mdp-greenhouse` simulator. One service, `~/get_tag_reading`, returns per-tag climate sensor readings. |
| `lupin_twin` | The digital twin: aggregates `/floranova/observations` into a latched `/twin/state` snapshot and an inverse-distance-weighted `/twin/get_field` heatmap service. |
| `lupin_hmi` | Teleop and the command bus: gamepad/keyboard driving, the `twist_mux` arbitration config, arm/gripper bridges, presets, calibration, and boot-time auto-home. |
| `lupin_web` | The browser mission console on `:8090` (Vite + React), talking to ROS over `rosbridge` (`:9090`), plus a 16-tool Gemini Live voice agent. No ROS nodes, it is a pure rosbridge client. |

Each package has its own page: [Navigation](/docs/navigation),
[Mission](/docs/mission), [Perception](/docs/perception),
[Digital twin](/docs/digital-twin), [Greenhouse bridge](/docs/greenhouse-bridge),

## The data-flow pipeline

Lupin turns an unmapped greenhouse into a queryable world model in one chain. The arrows
below are the actual topics and services that connect each stage:

<Mermaid chart={`
graph LR
  EXPLORE["EXPLORE<br/>mission BFS"]
  NAV["NAVIGATE<br/>Nav2 · MPPI"]
  PERC["PERCEIVE<br/>AprilTag + YOLO"]
  MISSION["MISSION SCAN<br/>orchestrator"]
  BRIDGE["greenhouse_bridge<br/>mdp-greenhouse sim"]
  TWIN["TWIN<br/>IDW field"]
  HMI["HMI / VOICE<br/>:8090 console"]

  EXPLORE -->|frontier goal| NAV
  NAV -->|/cmd_vel_auto| PERC
  PERC -->|/perception/discovered_tags| EXPLORE
  PERC -->|confirm_tag · KIND_FLOWER| MISSION
  MISSION <-->|GetTagReading| BRIDGE
  MISSION -->|/floranova/observations| TWIN
  TWIN -->|/twin/state · /twin/get_field| HMI
`} />

In prose:

1. **Explore.** `mission_orchestrator` runs a pure-Python BFS frontier planner over the
   live `/map` (from `slam_toolbox`) and drives toward unexplored cells until it has
   discovered *N* distinct AprilTags. `lupin_perception`'s `perception_aggregator`
   publishes the running registry on `/perception/discovered_tags` (`DiscoveredTags`),
   which the orchestrator counts toward `discovery_goal`.
2. **Navigate.** For each tag, the orchestrator synthesises an approach pose from table
   geometry and sends a `nav2_msgs/action/NavigateToPose` goal. Nav2's MPPI controller
   plans the path; its output never touches `/cmd_vel` directly (see the command bus
   below).
3. **Perceive.** While parked and SCANNING, `tag_annotator` detects the AprilTag and
   broadcasts a `tag_<id>` TF; the `yolo_detector` classifies tulip species and the
   `bug` anomaly on the gripper camera. On hardware, the orchestrator first calls
   `/perception/confirm_tag` (`ConfirmTag`) as a visual gate.
4. **Read + publish.** The orchestrator calls `/greenhouse_bridge/get_tag_reading`
   (`GetTagReading`) to fetch the tag's climate sensors, then publishes a typed
   `lupin_msgs/Observation` on **`/floranova/observations`**, one per closed tag. The
   aggregator independently publishes `KIND_FLOWER` observations on the same topic.
5. **Twin.** `lupin_twin` is the sole consumer that aggregates `/floranova/observations`
   into a per-tag world store, republishing the whole world on **`/twin/state`**
   (`TwinState`) and serving on-demand sensor heatmaps via `/twin/get_field` (`GetField`,
   inverse-distance-weighted).
6. **HMI.** The web console (`lupin_web`) subscribes to `/twin/state`, `/mission/state`,
   the camera streams, and the AprilTag overlay, and renders the world live, driveable
   by joystick, map-click, or voice.

<Callout type="info">
  `/floranova/observations` is the seam of the whole system. The mission orchestrator and
  the perception aggregator are the only publishers; the digital twin and the web HMI are
  the only consumers. Swapping the data source (sim oracle vs. real perception) leaves
  every downstream contract untouched.
</Callout>

## The command-arbitration bus

Autonomy, gamepad teleop, and the web HMI all want to drive the same base. Rather than
let them fight, every velocity source feeds a single `twist_mux` (configured in
`lupin_hmi/config/twist_mux.yaml`) that arbitrates by priority, higher wins, with a
per-source silence timeout:

| Input topic | Source | Priority | Timeout |
| --- | --- | --- | --- |
| `/cmd_vel_joy` | Xbox dead-man teleop (`teleop_twist_joy`) | 100 | 0.3 s |
| `/cmd_vel_manual` | Web HMI joystick / DS4 teleop | 50 | 0.5 s |
| `/cmd_vel_auto` | Nav2 (`velocity_smoother`) | 10 | 0.5 s |

The key design choice: **Nav2 never publishes to the controller directly.** Its
`controller_server` emits on `/cmd_vel_nav`, the `velocity_smoother` smooths it and
remaps the output to `/cmd_vel_auto`, the *lowest* priority input. So holding the Xbox
dead-man instantly preempts autonomy, and the web joystick overrides it too, all through
the same chain. The `twist_mux` output is remapped to the base controller's drive topic
(`cmd_vel_out`).

<Callout type="warn">
  The mux publishes its output **BEST_EFFORT**, because both the sim `twist_mux` and the
  real mecanum controller subscribe BEST_EFFORT, a RELIABLE publisher would silently
  drop every message on the QoS mismatch
  (`lupin_hmi/cmd_vel_mux.py`). The web HMI publishes to `/cmd_vel_manual` (priority 50),
  **not** to the controller topic, specifically so it shares this arbitration instead of
  racing Nav2.
</Callout>

## The sim ↔ hardware boundary

The repository keeps three long-lived branches (`main`, `sim`, `hardware`). The deep
point is how *little* changes between simulation and the real robot. Everything that
makes decisions is shared; only the edges differ.

**Identical across sim and hardware:**

- All mission logic, the frontier planner, and the state machine (`lupin_mission`).
- All perception nodes (`lupin_perception`), `tag_annotator` even runtime-selects the
  OpenCV ArUco API so the same node runs on OpenCV 4.6 and 4.7+.
- The digital twin (`lupin_twin`), it consumes only the abstract `Observation` stream,
  so it has zero dependency on Nav2 or Gazebo and behaves the same in both.
- The web HMI (`lupin_web`) and the entire `lupin_msgs` contract.
- Nav2 params: `base_frame`/`robot_base_frame` is `base_link`, and the odometry topic is
  `/mirte_base_controller/odom` (wheel-encoder odom) on both targets.

**What actually differs:**

| Concern | Simulation | Hardware |
| --- | --- | --- |
| Bring-up | `sim_full.launch.py` (Gazebo + everything) | `hardware.launch.py` (laptop) + onboard systemd |
| Drive topic | `/mirte_base_controller/cmd_vel_unstamped` | `/mirte_base_controller/cmd_vel` (stamped sink) |
| Localization | `slam_toolbox` owns `/map` + `map→odom` | AMCL against a saved map (or SLAM) |
| AMCL pose | seeded synthetically by `seed_amcl_pose` | published naturally by a running AMCL |
| Climate sensing | `mdp-greenhouse` oracle via the bridge | same bridge (real AprilTag IDs feed it) |
| Visual gate | `require_visual_confirmation` left `false` | `true` → `/perception/confirm_tag` runs |

<Tabs items={['Simulation','Hardware']}>
<Tab value='Simulation'>

One command brings up Gazebo, SLAM, Nav2, the bridge, the orchestrator, the twin, and the
web HMI, with an event-driven sentinel cascade (`/scan` → SLAM → `/map` → Nav2):

```bash
ros2 launch lupin_bringup sim_full.launch.py
```

The drive sink is `/mirte_base_controller/cmd_vel_unstamped`, and because there is no AMCL
in SLAM mode, `seed_amcl_pose` publishes one synthetic latched `/amcl_pose` to satisfy the
orchestrator's localization gate.

</Tab>
<Tab value='Hardware'>

The robot runs onboard systemd services on boot; the operator launches the rest from the
laptop with one command and per-subsystem flags:

```bash
ros2 launch lupin_bringup hardware.launch.py
```

The drive sink is the stamped `/mirte_base_controller/cmd_vel`. A real AMCL publishes
`/amcl_pose`, and the mission's `require_visual_confirmation` gate calls
`/perception/confirm_tag` before each scan.

</Tab>
</Tabs>

## Robot-side vs laptop-side split

Compute is deliberately split across the two machines. The robot keeps only what must be
local to drive, actuate, and stream cameras the moment it boots; everything heavy and
operator-facing runs on the laptop.

<Mermaid chart={`
graph TB
  subgraph ROBOT["ROBOT · Orange Pi · on boot"]
    direction TB
    ON["lupin-onboard.service<br/>twist_mux · arm_preset<br/>arm_calibrate · gripper_bridge"]
    CAM["lupin-cameras.service<br/>Orbbec + gripper cams (low FPS)<br/>web_video_server :8091 MJPEG"]
    HOME["lupin-auto-home.service<br/>self-heal controllers · home arm"]
  end
  subgraph LAPTOP["LAPTOP · hardware.launch.py"]
    direction TB
    WEB["lupin_web HMI + rosbridge"]
    SLAM["slam_toolbox (+ reset)"]
    NAV2["Nav2 · slam mode"]
    TWIN2["lupin_twin"]
    MIS["mission bundle<br/>bridge + orchestrator + amcl seed"]
    PERC2["lupin_perception<br/>tag + aggregator"]
    YOLO["YOLO detector · torch"]
    RVIZ["RViz"]
  end
  ROBOT ==>|WiFi · MJPEG + DDS discovery| LAPTOP
`} />

The three onboard units (`lupin-onboard`, `lupin-cameras`, `lupin-auto-home`) are
idempotent and `PartOf=mirte-ros`, so manual drive, arm, and gripper come up with the
robot independent of any laptop. The camera service is a deliberate CPU win: it relaunches
the cameras at config-driven low FPS on the *identical* vendor topic names and runs an
on-robot `web_video_server`, so only MJPEG, never raw RGB, crosses the WiFi link.
Details on the [Operations](/docs/operations) and [Networking](/docs/networking) pages.

## The latched-snapshot contract

Three of Lupin's status topics share a deliberate QoS pattern: **RELIABLE +
TRANSIENT_LOCAL** with depth 1 (latched). A late-joining subscriber, an HMI that just
loaded, or an orchestrator that just started a mission, immediately receives the current
state with no bootstrap round-trip. An empty array is a legitimate startup state, not an
error.

| Topic | Type | Publisher | Why latched |
| --- | --- | --- | --- |
| `/twin/state` | `lupin_msgs/TwinState` | `lupin_twin` (~1 Hz) | A freshly-connected HMI gets the whole observed world at once. |
| `/perception/discovered_tags` | `lupin_msgs/DiscoveredTags` | `perception_aggregator` (~2 Hz) | A mission that starts mid-run instantly sees the full tag registry. |
| `/mission/state` | `lupin_msgs/MissionState` | `mission_orchestrator` (5 Hz) | A late HMI sees the current lifecycle/phase snapshot immediately. |

`/floranova/observations` uses the same RELIABLE + TRANSIENT_LOCAL durability but with
depth 50, so a late subscriber replays the whole mission so far rather than just the last
reading.

<Callout type="warn">
  This pattern has a CLI gotcha. A native subscriber that defaults to VOLATILE durability
  (e.g. a plain `ros2 topic echo`) will see *nothing* on first connect, because the
  publisher is TRANSIENT_LOCAL. The web HMI consumes these via rosbridge and is
  unaffected; from the CLI, match the durability explicitly.
</Callout>

The contract is also encoded in the message design itself: `Observation` is a polymorphic
envelope keyed by `kind` (`KIND_TAG_READING` / `KIND_FLOWER` / `KIND_ANOMALY`), and a
`geometry_msgs/Pose` with `orientation.w == 0` is the agreed in-band sentinel for "this
pose was never observed", so consumers treat it as missing without a separate validity
flag. See [Interfaces](/docs/interfaces) for the full message and service definitions.
